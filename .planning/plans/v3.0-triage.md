## Candidate A: AiGenerated Provenance Flag

> Track whether a keyword/category was created by AI tagging vs. manual entry.

### Current State

| Layer | Status | Details |
|-------|--------|---------|
| **Data model** | ❌ No field | `Keyword` has `Id, Name, NormalizedName, ParentId, UsageCount`. No provenance column. Same for `Category`. |
| **Service layer** | ❌ No flag path | `KeywordService.AssociateKeywordsAsync()` creates keywords unconditionally. `AiTaggingService.TagAssetAsync()` calls it without provenance. |
| **Wire protocol** | ❌ No transport | `KeywordInfo` / `CategoryInfo` protobuf messages have no flag. `AssetDetail.Tags` is `List<string>` — no metadata per tag. |
| **UI** | ❌ No badge | Tag chips have no visual distinction for AI-generated vs. manual. |
| **Confidence** | ⚠️ Rank heuristic | Current confidence is estimated from output order (top 25% → 0.9+, middle 50% → 0.6-0.9, bottom 25% → 0.3-0.6). Not model logits. |

### Scope of Work

1. **Data model** — Add `bool IsAiGenerated` to `Keyword` and `Category` entities. EF Core migration.
2. **KeywordService** — Add `bool isAiGenerated = false` parameter to `AssociateKeywordsAsync()`. Save the flag.
3. **AiTaggingService** — Pass `isAiGenerated: true` in `TagAssetAsync()`. Read-only analyze path (`AnalyzeAssetWithScoresAsync`) is already detached from DB writes.
4. **Wire protocol** — Add `IsAiGenerated` to `KeywordInfo` / `CategoryInfo` protobuf messages. Bump field numbers.
5. **AssetDetail** — When tags are returned as `List<string>`, the provenance info is lost. Either (a) add a parallel `List<bool> TagsAreAiGenerated` field, or (b) switch to a richer tag DTO. Both are wire-breaking changes.
6. **Review dialog** — The `TagAssetAsync()` path (Trigger A / ingest) writes directly to DB. The review dialog path (Trigger B) uses `AnalyzeAssetWithScoresAsync()` which is read-only — tagging happens via manual accept/reject. Provenance would need to be set when the user clicks "Apply" in the review dialog.
7. **UI** — Add small "AI" badge or sparkle icon next to AI-generated keywords in the metadata editor tag list. Gray/dimmed styling.
8. **Batch editing** — Consider whether batch operations should distinguish provenance (e.g., "don't overwrite AI tags with manual values").
9. **Category service** — Mirror keyword changes for `Category` and `CategoryService`.
10. **Tests** — 8-12 new tests covering flag persistence, wire serialization, UI badge visibility.

### Triage Assessment

| Dimension | Score | Commentary |
|-----------|-------|------------|
| **User value** | 🟡 **Medium** | Useful for power users who want to know which tags came from AI. Most users won't notice or care until they need to audit AI contributions. |
| **Effort** | 🟢 **Low-Medium** | ~6-8 files, ~150 LOC. Straightforward bool column add. The main risk is the wire protocol change (backward compat). |
| **Risk** | 🟢 **Low** | Wire-breaking change if using `List<string>` Tags. Can mitigate with parallel `List<bool>` field. EF migration is additive. |
| **Dependencies** | 🟢 **None** | Standalone feature. No blocker from other v3 candidates. |
| **When it adds value** | 🟡 **Post-audit** | Value only shows when someone audits "which tags are trustworthy." Low urgency. |

**Recommendation: 🟢 v3.0 candidate — accept if wire-breaking is acceptable.**

If wire compatibility with existing v1.x clients is a requirement, defer to v3.1 with a new message version negotiation scheme.

## Candidate B: Multi-User Batch Editing

> Enable batch metadata editing when connected to a shared BrokerService (PostgreSQL/SQL Server).

### Current State

| Layer | Status | Details |
|-------|--------|---------|
| **Client batch code** | ⛔ Hard-gated | `PropertyInspectorViewModel.ApplyBatchEditAsync()` returns early if `!_modeManager.IsStandalone`. Same for `DetectBatchMixedValuesAsync()`, `ComputeAggregatedTagsAsync()`, etc. |
| **Broker single update** | ✅ Working | `AssetHandler.UpdateAssetAsync()` handles per-asset update with version conflict detection, change notification broadcast, metadata writeback. Message type `UpdateAssetRequest` (opcode 16). |
| **Broker batch pattern** | ✅ Exists | `BulkPermanentDeleteAssetAsync()` shows the pattern: `List<string> Ids` in request, `for` loop over assets, single `SaveChangesAsync()`, per-asset change notifications. |
| **Wire protocol** | ❌ No batch message | No `BatchUpdateAssetRequest` / `BatchUpdateAssetResponse` message type. `UpdateAssetRequest` is single-asset. |
| **Version conflict** | ✅ Single-asset | `UpdateAssetAsync()` checks `req.ExpectedVersion > 0 && asset.Version != req.ExpectedVersion`. Batch would need a conflict resolution strategy. |

### Scope of Work

1. **New protobuf message** — Create `BatchUpdateAssetRequest` with:
   - `List<string> Ids` — which assets
   - `string Title` / `string Description` / `int Rating` / etc. — common values to apply
   - `List<int> FieldFlags` — bitmap of which fields to update (allows mixed batch where only specific fields are set)
   - `int ExpectedVersion` — optional; if > 0, all assets must match
2. **New protobuf response** — Create `BatchUpdateAssetResponse` with:
   - `int UpdatedCount`
   - `List<BatchConflictInfo> Conflicts` — per-asset version conflicts
   - `int TotalRequested`
3. **New message opcodes** — Add `BatchUpdateAssetRequest = 102` and `BatchUpdateAssetResponse = 103` to `MessageTypeCode`
4. **Broker handler** — `AssetHandler.BatchUpdateAssetsAsync()`:
   - Authorize `asset:update`
   - Load all assets by IDs
   - Check versions if `ExpectedVersion > 0`
   - Apply field updates based on `FieldFlags` bitmap
   - Single `SaveChangesAsync()` for atomicity
   - Broadcast change notifications per asset
   - Return conflict details
5. **Wire routing** — Add dispatch in `ConnectionHandler` for opcode 102
6. **Client refactoring** — Remove `!_modeManager.IsStandalone` guards from `PropertyInspectorViewModel`:
   - `ApplyBatchEditAsync()` — route through `BrokerClient.SendAsync()` instead of direct `AppDbContext`
   - `DetectBatchMixedValuesAsync()` — needs a new broker message or batch `GetAssetBatchRequest` to fetch multi-asset metadata
   - `ComputeAggregatedTagsAsync()` / `ComputeAggregatedCategoriesAsync()` — same
   - Tag/category autocomplete — already uses `_modeManager.IsStandalone` in `LoadTagAutoCompleteSourceAsync()` and `DetectBatchMixedValuesAsync()`
7. **Version conflict UX** — Show conflicts after batch apply with option to retry or accept latest
8. **Tests** — Full suite: broker handler (unit + integration), client ViewModel, conflict scenarios, atomicity, mixed results

### Triage Assessment

| Dimension | Score | Commentary |
|-----------|-------|------------|
| **User value** | 🔴 **Low-Medium** | Multi-user mode is used by a minority of deployments (requires BrokerService + external DB). Of those, batch editing is a nice-to-have over single-asset editing which already works. |
| **Effort** | 🔴 **High** | ~10-14 files, ~400-500 LOC. New protobuf messages, broker handler, client ViewModel refactoring (3 methods need multi-user paths), conflict UX, tests. |
| **Risk** | 🟡 **Medium** | Version conflict handling in batch is inherently complex (partial failures, retry semantics). Wire protocol change is additive (new opcodes), no backward compat issue. |
| **Dependencies** | 🟡 **Medium** | Needs existing multi-user setup (BrokerService + PostgreSQL/SQL Server). Cannot be tested without Docker (2 existing tests are already skipped). |
| **When it adds value** | 🟡 **Post-deployment** | Only valuable to teams using shared database mode. Single-user batch already works in standalone. |

**Recommendation: 🟡 v3.0 candidate — defer to v3.1 or later unless there is explicit user demand.**

The high effort-to-value ratio makes this a poor v3.0 investment. The single-asset update path already works in multi-user mode, which covers the 90% case for shared deployments.
