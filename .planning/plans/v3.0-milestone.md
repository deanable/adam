## Phase 18: Integration Foundation — Plugin System

**Goal:** Fulfill INTG-V2-01 — create a plugin system for third-party metadata extractors so the catalog can be extended without modifying core code.

### Tasks

#### T18.1 — Plugin Contract Interface
- Define `IMetadataExtractorPlugin` interface in `Adam.Shared.Plugins`:
  - `string Name { get; }` — plugin display name
  - `string[] SupportedExtensions { get; }` — file extensions this plugin handles (e.g., `[".cr3", ".hif"]`)
  - `Task<ExtractedMetadata> ExtractAsync(string filePath, CancellationToken ct)` — returns metadata
  - `int Priority { get; }` — plugin priority (higher = preferred over built-in for same extension)
- Define `ExtractedMetadata` DTO — Title, Description, Keywords, Categories, Rating, Label, Flag, GPS, Copyright, CameraMake/Model, DateTaken, etc.
- **Files:** `src/Adam.Shared/Plugins/IMetadataExtractorPlugin.cs` (new), `src/Adam.Shared/Plugins/ExtractedMetadata.cs` (new)

#### T18.2 — Plugin Loader
- `PluginLoaderService`:
  - Scans `./plugins/` directory for assemblies matching `*.Plugin.dll`
  - Uses `AssemblyLoadContext` for isolated loading (can unload)
  - Loads all types implementing `IMetadataExtractorPlugin`
  - `ListPlugins()` — returns loaded plugins
  - `GetExtractorForExtension(string ext)` — returns highest-priority plugin matching extension, or null (fall back to built-in)
- **Files:** `src/Adam.Shared/Services/PluginLoaderService.cs` (new), `src/Adam.Shared/Adam.Shared.csproj` (no new deps needed — uses built-in `AssemblyLoadContext`)
- **Tests:** 5-8 tests for loading, priority ordering, isolation, missing directory handling

#### T18.3 — Integration with Ingestion Pipeline
- Modify the ingestion/extraction pipeline to query `PluginLoaderService.GetExtractorForExtension()` before falling back to built-in `MetadataExtractor` / `OfficeDocumentExtractor` / image analysis
- Log which plugin was used for each asset type
- **Files:** `src/Adam.Shared/Services/MetadataExtractorService.cs` (or equivalent)
- **Tests:** 3-4 tests for plugin vs built-in fallback logic

#### T18.4 — Sample Plugin Project
- Create `samples/plugins/SampleMetadataPlugin/` with a basic implementation
- Demonstrate: register for `.xmp` files (extract from sidecar XML), or `.cr3` (Canon RAW v3)
- Include `README.md` explaining how to build and install plugins
- **Files:** `samples/plugins/SampleMetadataPlugin/` directory with project + code
- **Tests:** Verify plugin loads and returns metadata

#### T18.5 — Plugin Management UI
- Minimal "Plugins" settings page in sidebar or settings dialog
- List loaded plugins with name, extensions handled, status (loaded/error)
- Enable/disable toggle per plugin
- **Files:** Likely integration into existing settings/sidebar in `Adam.CatalogBrowser`

### Success Criteria
- ✅ Plugin interface defined with clear contract
- ✅ Plugins load from `./plugins/` directory at startup
- ✅ Plugin metadata replaces or augments built-in extraction per extension
- ✅ Sample plugin project demonstrates the pattern
- ✅ Plugin loading is isolated (unloadable, no version conflicts)
- ✅ 15+ new tests
