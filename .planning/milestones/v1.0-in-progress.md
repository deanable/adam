# Milestone v1.0-in-progress: Core DAM

**Status:** 🚧 In Progress — Phases 1-2 Complete, Phase 3 Pending
**Date:** 2026-05-23
**Phases:** 1-2 of 7

## Overview

The v1.0 milestone delivers a working digital asset catalog in both standalone and multi-user modes. Phases 1 and 2 are complete; Phase 3 (Multi-User Concurrency) remains.

## Completed Phases

### Phase 1: Standalone Mode ✅

**Goal:** A user launches the catalog browser with zero external dependencies, selects a folder of assets, and can browse, search, and view metadata.

**Key Deliverables:**
- Avalonia catalog browser with grid, loupe, and compare views
- SQLite-based local asset catalog with full-text search
- EXIF, IPTC, and XMP metadata extraction and display
- Hierarchical keyword tagging, star ratings, color labels, and flagging
- Curated collections independent of disk folder structure
- SHA256 duplicate detection during folder scan

**Status:** All 11 CATA requirements and META-05 delivered.

### Phase 2: Multi-User Foundation ✅

**Goal:** The catalog browser connects to a running broker service over TCP, authenticates, and browses the shared catalog.

**Key Deliverables:**
- TLS transport with self-signed dev cert fallback (T2.1)
- JWT secret via environment variable + quickstart docs (T2.2)
- Authorization middleware on all handlers (T2.3)
- Stable opcode enum routing (T2.5) — fixed string-based MessageType bug (T2.4)
- BrokerClient auto-reconnect with exponential backoff (T2.6)
- Sidebar fully functional in multi-user mode (T2.7):
  - Folder tree with asset counts
  - Keyword hierarchy with parent/child propagation
  - Media format counts (Image, Video, Document, Audio)
  - Metadata categories hierarchy
  - Date taken tree (year → month)
- Graceful shutdown with task observation (T2.9)
- Brute-force protection via LoginRateLimiter (T2.13)
- JWT jti + iat claims for replay protection (T2.14)
- Structured security logging (T2.15)
- EF Core optimistic concurrency with Version token (T2.16)
- AsNoTracking + AsSplitQuery for read queries (T2.17)
- Connection resiliency with EnableRetryOnFailure (T2.18)
- Client-side timeout/retry on SendAsync (T2.20)
- ChangePoller retry with token expiry detection (T2.21, T2.23)
- Connection status UI in MainWindow (T2.22)
- FolderWatcherHostedService with debounced FileSystemWatcher (T2.24)
- Watched folder DB persistence + admin panel UI (T2.25)

**Architecture Review Blockers Resolved:**
- ✅ CRITICAL-4: No TLS → TLS implemented
- ✅ CRITICAL-5: Hardcoded JWT secret → env var required
- ✅ CRITICAL-6: No authorization → middleware on all handlers
- ✅ CRITICAL-8: Raw tag bug → fixed
- ✅ CRITICAL-9: String routing → opcode enum
- ✅ CRITICAL-10: No reconnection → auto-reconnect implemented
- ✅ HIGH-13: Sidebar non-functional → fully wired

**Critical Issues Fixed:**
- Port mismatch (BrokerClient now uses 9100)
- Static mutable JWT key (instance field)
- TCS memory leak in BrokerClient

## Pending Phase

### Phase 3: Multi-User Concurrency

**Goal:** Multiple users access the broker simultaneously with consistent state and real-time change propagation.

**Requirements:** BROK-02, BROK-03, BROK-04

**Success Criteria:**
1. 10 concurrent users browse and search with all responses under 3 seconds
2. Metadata change by User A appears for User B within 5 seconds
3. Simultaneous edits to same asset resolve with last-write-wins; both see final state

## Key Decisions

| Decision | Outcome |
|----------|---------|
| Manual protobuf serialization | Confirmed — no protoc generator in use |
| TLS required for multi-user | Implemented with dev self-signed cert fallback |
| Opcode enum for routing | Values 1-99 allocated, 100+ reserved |
| Dual-mode architecture | Standalone first, multi-user via BrokerClient |
